## Description

Surfaced by the v0.17 fresh-eyes audit (same bug class as the WhatsApp allowlist no-op fixed in 0d9e026). Pre-existing since v0.13; NOT v0.17-introduced.

Two issues:
1. GatewayAllowlistKind.kind(for:) maps "google-chat"/"googlechat" → .channels → writes `<p>.allowed_channels`, but v0.17 google_chat gates via the `GOOGLE_CHAT_ALLOWED_USERS` env / allowed_users (plugins/platforms/google_chat/adapter.py:3320), NOT allowed_channels. So Scarf's google-chat allowlist is a silent no-op — drop it from the chat-id editor (like WhatsApp) or rewire to allowed_users.
2. Canonical platform id changed to `google_chat` (underscore) in v0.17 — Platform("google_chat"), plugin.yaml name google_chat-platform (the long-standing TODO(WS-5-Q1) about the spelling). Scarf still uses "google-chat" (hyphen) in KnownPlatforms.all + KnownPlatforms.icon + PlatformsView googleChatPanel + GatewayAllowlistKind. Audit/verify the full google_chat surface against v0.17 and reconcile the identifier (roster, icon, dispatch, allowlist) so config writes hit the keys Hermes reads.

Scope note: broader than the v0.17 catch-up (it's a pre-existing platform-id drift), so tracked separately rather than partial-fixed inline.
