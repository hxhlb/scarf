# Scarf v2.9.1 / v2.8.0 - 100% CPU usage (single core) when connected to Hermes

## Description

Scarf consistently pegs a single CPU core at 100% after launching, regardless of whether the Hermes gateway is running. This was observed on both v2.8.0 and v2.9.1.

## Environment

- macOS: 26.4.1
- Hermes: v0.14.0 (2026.5.16)
- Scarf: v2.9.1 (also reproduced on v2.8.0)
- Session count: ~477
- State DB: ~245MB

## Steps to Reproduce

1. Launch Scarf
2. Observe CPU gradually climb to 100% on a single core within 30-60 seconds
3. App remains functional but system feels sluggish

## Observations

- CPU stays at ~100% indefinitely, never settles
- Hermes gateway connection status does NOT affect CPU usage
- Restart temporarily drops CPU to ~15-20% for ~15-30 seconds, then climbs back to 100%
- Zero TCP connections established when spinning
- 100% reproducible across restarts

## Log Signals

From `log show --predicate 'process == "scarf"'`:

- Repeated `NSSceneFenceAction` updates suggesting constant UI/compositor invalidation
- Frequent `NSPersistentUIManager flushAllChanges` calls (state restoration saving every ~2-4 seconds)
- `XPC_ERROR_CONNECTION_INTERRUPTED` observed when gateway unreachable
- No error-level log messages during the spinning

## Resource Usage at Peak

- CPU: 100% (single core)
- Memory: ~145MB RSS
- Open FDs: ~141
- No TCP connections

## Suspected Cause

Frequent `NSPersistentUIManager flushAllChanges` + constant SceneFence updates suggest a SwiftUI render loop. Some published state or binding is continuously invalidating the view hierarchy. Possible candidates:
- A data source (session list, skills tree, cron) publishing updates in a tight loop
- A file watcher or timer firing too often
- Scene state restoration writing persistently

## Workaround

None found. Must quit the app when not actively in use.
