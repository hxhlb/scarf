## Executive summary
The dominant real risk is **state churn on view re-entry**, not raw main-thread blocking: the app is largely disciplined about pushing remote/subprocess work off-main via `Task.detached`, so most findings are allocation churn or redundant re-fetches rather than UI freezes. The one genuine hot-path defect is in chat history loading — a v0.11 schema/column-index mismatch in `HermesDataService` that crashes or silently corrupts `reasoning_content` on a common path. A high refute rate (11 of 23 candidates refuted) is expected and healthy for a perf pass.

- **Counts (authoritative, programmatic):** **1 critical** / **1 high** / **5 medium** / **5 low** — 12 confirmed, **11 refuted**.
- **Note:** the critical was *escalated by the adversarial verifier* — the first-pass finder reported the two-phase reasoning load as "already correct"; the verifier opened the files and proved it broken.

## Critical & high-severity findings (prioritized)

- **[CRITICAL] Chat `reasoning_content` two-phase load is broken (column-index mismatch)** — `scarf/Packages/ScarfCore/Sources/ScarfCore/Services/HermesDataService.swift:238-292` (also `messageFromRow` ~1154, `fetchReasoningContent` ~493) — *dimension: external-dispatch.* On every chat resume against a v0.11+ schema with reasoning, `messageColumnsSkeleton` emits an 11-column row (indices 0–10) but `messageFromRow` unconditionally reads index 11 when `hasV011Schema` is true → out-of-bounds crash or silently wrong data. Worse, the intended lazy fetch never fires: `RichMessageBubble` renders `message.preferredReasoning` directly and has no `.task`/`.onChange` to call `fetchReasoningContent`, which exists (`:493-503`) but is **never called by the chat UI**. **Fix:** make `messageFromRow` index-aware of skeleton rows (guard the index-11 read), and actually trigger `fetchReasoningContent` when the reasoning disclosure opens. *(The two-phase design — excluding `reasoning_content` from the first fetch to mitigate issue #74 payload bloat — is the right instinct; only the execution is buggy.)*

- **[HIGH] Project Kanban tab does synchronous file I/O on tab switch** — `scarf/scarf/Features/Projects/Views/ProjectsView.swift:454-481` (→ `ProjectKanbanTab.resolveTenant` ~L52 → `KanbanTenantResolver.resolveOrMint`) — *dimension: nav-cascade.* The project dashboard `switch` on `selectedTab` destroys/recreates tab views; when the Kanban tab first appears, `resolveTenant()` calls `KanbanTenantResolver.resolveOrMint()` inline in `.task` context, iterating all projects doing synchronous `FileManager.fileExists/readFile/writeFile` on the main thread — blocking I/O on a common tab-switch interaction (idempotent, so once per project, but still inline on main). **Fix:** move `resolveOrMint()` off-main (`Task.detached`) and surface a loading state while the tenant resolves.

## Medium-severity findings (grouped by dimension)

**main-actor-blocking**
- `scarf/scarf/Features/Health/ViewModels/HealthViewModel.swift:704-758` — `stopDashboard()` runs `lsof` + `waitUntilExit()` synchronously on main from a Button action (100ms–1s+ freeze on the rare "Stop Dashboard" click) — wrap `dashboardListenerPID()` in `Task.detached`, sync back to update UI.

**external-dispatch**
- `scarf/scarf/Features/Health/ViewModels/HealthViewModel.swift:144-176` vs `HealthView.swift:124-128` — `load()`'s detached task handle isn't stored, so tabbing away mid-load keeps burning 6+ SSH round-trips — store the handle and add `.onDisappear { viewModel.cancelLoad() }` (mirror `DashboardViewModel.inFlightLoad`).

**body-churn**
- `scarf/scarf/Features/Kanban/Views/KanbanCardView.swift:395-405` — `relativeShort()` allocates a fresh `ISO8601DateFormatter` per card per render (20–30+ cards in a `LazyVStack`, re-rendered on the 10s poll) — promote to `static let` (the file already has a cached `relativeFormatter` at 407–411).
- `scarf/scarf/Features/Sessions/Views/SessionsView.swift:604-606` — `updatedLabel` allocates a `RelativeDateTimeFormatter` per row, re-fired by per-row hover `@State` — cache as `static let`.

**nav-cascade**
- `ContentView.swift:50-84` + 9 feature views (`SettingsView.swift:108`, `HealthView.swift:124-127`, Platforms, Plugins, QuickCommands, Models, MCPServers, Webhooks, Cron) — the `@ViewBuilder` switch on `coordinator.selectedSection` destroys/recreates each subtree, so `.onAppear { viewModel.load() }` re-fires multi-call remote loads on every re-entry — switch to `.task` (as `SessionsView`/`ChatView` already do). *Note: off-main, so no beachball; the cost is redundant SSH round-trips on remote.*

## Recurring patterns (memory-note candidates)

1. **Formatter allocation inside SwiftUI computed properties.** Date/relative formatters are constructed inside computed properties / per-row helpers that SwiftUI re-evaluates per render and per-row state change (e.g. hover), instead of a cached `static let`. The codebase already has the right pattern (`KanbanCardView.swift:407-411`) but applies it inconsistently. Where: `KanbanCardView.swift:395-405`, `SessionsView.swift:604-606`, `ProjectSessionsView.swift:186-189`, `ModelPickerSheet.swift:415-417`.
   → **Suggested note:** "Cache formatters as static let, never in view bodies" (`conventions/`).
2. **`.onAppear { viewModel.load() }` on views destroyed by `@ViewBuilder` switches.** Navigation uses `switch` on a selected-section/-tab enum that destroys+recreates subtrees per selection, so `.onAppear`-triggered loads re-fire on every re-entry, re-running multi-call remote fetches. Use `.task` (fires once per instance, auto-cancels) as `SessionsView`/`ChatView` already do; for true state persistence, hoist the view or cache in the coordinator. Where: `ContentView.swift:50-84` + 9 feature views, `ProjectsView.swift:454-481`.
   → **Suggested note:** "Prefer .task over .onAppear for view-load fetches behind switch-based navigation" (`architecture/`).
3. **Off-main work without a stored, cancellable task handle.** Long remote/subprocess work is correctly dispatched off-main, but the handle is often not stored, so it can't be cancelled when the user navigates away mid-flight — leaving SSH round-trips / remote processes running. `DashboardViewModel.inFlightLoad` is the reference; `HealthViewModel.load()` the counter-example. Where: `HealthViewModel.swift:144-176`, `BackupServerSheet.swift:31-34`.
   → **Suggested note:** "Store cancellable handles for off-main remote work; cancel on disappear" (`conventions/`).

## Healthy areas
- **Two-phase chat message fetch (intent correct).** `HermesDataService` deliberately excludes `reasoning_content` from the first fetch to mitigate issue #74 payload bloat — right architecture for large reasoning blobs (execution bug flagged above).
- **Consistent off-main dispatch for remote/subprocess work.** Health, Settings, and feature views run multi-call remote loads under `Task.detached`; `ScarfApp.init` defers startup work off-main (`scarfApp.swift:75-124`) — so even the redundant-fetch findings don't manifest as beachballs.
- **Cancellation/guard building blocks already exist.** `DashboardViewModel.inFlightLoad` guards overlapping loads; `RemoteBackupService` threads `Task.checkCancellation()` through its stages (lines 214/237/260/290/332/382); `SessionsView`/`ChatView` use `.task` correctly.
