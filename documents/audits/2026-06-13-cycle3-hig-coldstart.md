## Executive summary

The dominant gap is **incomplete async-feedback affordances** тАФ several primary panes fire SSH data loads via `.task`/`.onChange` without surfacing a loading overlay, even though the project ships a reusable `.loadingOverlay()` modifier and uses it correctly across most panes (so these are *consistency* gaps, not missing infrastructure). ## Medium-severity findings (grouped by dimension)

### cold-start  *(severity medium, but high practical priority тАФ gh#102-adjacent)*
- `scarf/scarf/scarfApp.swift:346-515` тАФ **macOS never pauses `ServerLiveStatus` polling when backgrounded.** `startPolling` runs an unbounded 10s `while` loop with no `NSApplication.didResignActiveNotification` listener, so polling (and 5тАУ8s SSH timeouts against unreachable remotes, per open window) continues when all windows are minimized or the app loses focus. macOS only listens for `didBecomeActiveNotification` (capability refresh), never resign-active. iOS already does this correctly via scene-phase teardown. **This is the static fingerprint of the gh#102 "100% CPU on idle connection" report.** **Fix:** add a `didResignActive`/`didBecomeActive` pair routed to a central pause/resume, with a `paused` gate in the poll loop.

## Recurring patterns (memory-note candidates)

1. **Inconsistent adoption of the `.loadingOverlay()` async-feedback affordance.** The project ships `.loadingOverlay(isLoading, label:isEmpty:)` and uses it correctly across many panes (Dashboard, Activity, Settings, Memory, Plugins, Health, CredentialPools, Cron, MCPServers), but several SSH-loading panes omit it тАФ and some VMs (Logs, Sessions) don't even expose `isLoading`. Where: `InsightsView.swift:21-47` (high), `SessionsView.swift:43-77`, `LogsView.swift:17-28`.
   тЖТ **Suggested note:** "Every data-loading pane must drive `.loadingOverlay()`" (`conventions/`).
2. **macOS app lifecycle parity with iOS scene-phase handling.** iOS implements full scene-phase pause/resume (`ScarfGoCoordinator.setScenePhase`, `ChatController.handleScenePhase`); macOS only observes `didBecomeActiveNotification` and never reacts to losing focus, so background work (`ServerLiveStatus` polling) keeps running. Where: `scarfApp.swift:346-515` vs iOS `ScarfIOSApp.swift:164` + `ChatView.swift:234-237`.
   тЖТ **Suggested note:** "macOS must mirror iOS scene-phase pause/resume for background work" (`architecture/`).
3. **Standard macOS menu-bar conventions (тМШ, / Help / тМШF) are under-served.** Functionality exists in-app (sidebar Settings, search fields) but isn't reachable via the reflexive keyboard/menu affordances. Where: `scarfApp.swift:207-218`, `SessionsView.swift:253-280`.
   тЖТ **Suggested note:** "Wire standard macOS menu-bar commands (Settings/Help/Find)" (`conventions/`).
